# 架构说明

## 总体架构

系统采用分布式黑板风格：

- 黑板：Redis，保存地图、小车状态、路径队列、用户、回放帧等共享数据。
- 消息总线：RabbitMQ，负责 Controller 与知识源之间的调度消息。
- 知识源：Car、Navigator、Recorder，均可独立进程部署。
- 控制器：Controller，是唯一调度器，按节拍驱动系统。
- 展示端：现有 Swing `View`，只负责配置、显示和回放操作。

## 组件职责

| 组件 | 职责 |
|---|---|
| View | 登录、地图配置、小车/障碍物添加、算法选择、实时地图显示、回放界面 |
| Controller | 监听开始/停止命令，扫描小车状态，派发导航任务，广播移动节拍 |
| Navigator | 为小车选择未探索目标，使用路径算法生成路径队列 |
| Car | 按节拍读取路径、移动、避障、点亮 3x3 视野 |
| Recorder | 保存探索过程帧，回放时恢复帧到 Redis |
| Tools | 初始化账号、队列和运行数据 |

## Redis 数据字典

| Key | 类型 | 说明 |
|---|---|---|
| `map_width` / `map_height` | String | 地图宽高 |
| `MapView` | Bitmap | 已探索区域，1 表示已探索 |
| `blockview` | Bitmap | 障碍物区域，1 表示障碍 |
| `Algorithm` | String | `0=A*`，`1=Bidirectional A*`，`2=Dijkstra` |
| `Cars:1..5` | Hash | 小车坐标、终点、方向和状态 |
| `1_task_queue..5_task_queue` | List | 小车路径队列，元素为 `x,y` |
| `Users:<name>` | Hash | 登录用户密码哈希和角色 |
| `Save` | Hash | 回放记录数量、当前记录、速度和进度 |
| `Record:<file>:<frame>` | Hash | 探索回放帧 |

## RabbitMQ 协议

| 交换机/队列 | 类型 | 说明 |
|---|---|---|
| `controller` | Fanout | View 发布开始/停止命令 |
| `controller.start` | Queue | Controller 消费开始/停止命令 |
| `navigator.start` | Queue | 与 View 兼容保留 |
| `save` | Fanout | View 发布记录/回放命令 |
| `save.start` | Queue | Recorder 消费记录/回放命令 |
| `car.broadcast` | Fanout | Controller 广播移动节拍 |
| `car.no1..5` | Queue | 每辆小车独立消费节拍 |
| `navigator.messagesent` | Direct | Controller 向导航器派发任务 |
| `navigator.no1..4` | Queue | 导航器工作队列 |

## 控制流程

1. View 写入地图、小车、障碍物和算法配置到 Redis。
2. View 点击“开始探索”，向 `controller` 和 `save` 发送 `1`。
3. Controller 开始节拍循环。
4. Controller 查找空闲小车，向空闲 Navigator 发送导航任务。
5. Navigator 读取地图和探索状态，选取未探索目标，将其他小车当前位置视为动态障碍，生成路径并写入任务队列。
6. Controller 广播节拍。
7. Car 收到节拍后移动一步，移动前再次检查静态障碍和其他小车占位，更新 `Cars:*` 并点亮 `MapView`。
8. Recorder 周期性保存 Redis 快照。
9. 探索率达到目标后 Controller 停止，Recorder 停止记录。
10. Analyst 回放时，Recorder 按 `Save` 字段恢复历史帧。

## 实验要求对应关系

- 各组件共享数据：通过 Redis 黑板完成。
- 各组件高度独立：Controller、Navigator、Car、Recorder 均有独立入口。
- 易于升级：算法通过 `PathFinder` 接口扩展。
- 运行效率：地图使用 bitmap，路径使用 Redis list，小车按节拍增量移动。
- 易于调试：统一配置、运行说明和一键启动器。
- 界面组件：沿用现有 Swing View。
