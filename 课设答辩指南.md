#### QA-01 你们为什么选择黑板架构？

**Q**（王老师）：你们这个系统为什么要用黑板架构？用普通的 RESTful 微服务行不行？

**A**：
> 我们选择黑板架构主要基于三个考虑：
>
> **第一，问题本质是数据驱动协作。** 多车巡检的核心需求是所有小车共享同一张地图、实时看到彼此的探索进度——这不是典型的请求-响应场景，而是"所有知识源盯着同一块数据做决策"。黑板架构天然适合这种模式：Redis 作为黑板存储全局状态，各知识源独立读取和写入。
>
> **第二，知识源天然独立。** 路径规划（Navigator）和小车移动（Car）是两种完全不同性质的工作——前者是 CPU 密集型搜索，后者是 I/O 密集型的 Redis 读写+MQ 消费。黑板架构允许它们各自独立开发、独立部署、独立扩缩容。
>
> **第三，课设的增量开发要求。** 课设要求系统能不断改进——黑板架构新增一个知识源只需"读写 Redis + 监听 MQ"，不影响任何现有组件。例如我们要加一个"障碍物动态检测"功能，只需要加一个新模块监听障碍物数据，完全不需要改 Controller 或 Car。
>
> RESTful 微服务当然也能实现，但会引入服务发现、负载均衡、请求链路追踪等额外复杂度——对于课设场景，黑板架构更简洁、更匹配问题本质。

**由**：A（组长）

#### QA-02 Redis 和 RabbitMQ 分别解决什么问题？

**Q**（王老师）：能不能讲一下为什么既用 Redis 又用 RabbitMQ？单一组件不能解决吗？

**A**：
> 它们解决不同层面的问题：
>
> **Redis 解决"数据共享"问题**——它作为黑板存储全局状态。所有组件需要共享的数据（地图探索状态、小车位置、障碍物布局）都放在 Redis 里。Redis 的 Bitmap 适合存储地图位图、Hash 适合存小车信息、List 适合存路径队列、SETNX 能实现分布式锁——这些数据结构恰好匹配我们的需求。
>
> **RabbitMQ 解决"消息协调"问题**——它作为消息总线负责组件间的控制流。Controller 需要广播节拍命令给所有小车（Fanout）、精确派发导航任务给特定 Navigator worker（Direct）——RabbitMQ 的交换器类型恰好支持这些模式。
>
> 单一组件做不到：Redis 的 Pub/Sub 不保证消息可靠投递（订阅者离线会丢消息），不适合控制命令；RabbitMQ 不适合做位图存储和分布式锁。两者分工明确、互补。

**注意**：可以用"Redis 是共享黑板，RabbitMQ 是传令兵"这个比喻让老师快速理解。

**由**：A（组长）

#### QA-03 组件之间到底怎么通信的？画一下数据流？

**Q**（王老师）：你讲一下 Controller 和 Car、Navigator 之间具体怎么传数据的？

**A**：
> 好的，我以一次完整的"Controller 发出节拍、小车移动一步"为例说明：
>
> **Step 1（Controller → Redis）**：Controller 的 tick 循环开始，从 Redis 读取所有 `Cars:*` 键，获取在线小车列表。
>
> **Step 2（Controller → RabbitMQ → Navigator）**：对每辆空闲小车，Controller 轮询 `navigator_status` 找到空闲的 Navigator worker，通过 RabbitMQ Direct Exchange 向 `navigator.no{X}` 队列发送导航任务 JSON（包含小车 ID、当前位置）。
>
> **Step 3（Navigator → Redis）**：Navigator worker 收到任务后，从 Redis 读取 `MapView` 和 `blockview` 位图，BFS 选目标，A* 搜路径，然后 `RPUSH` 到小车 `{carId}_task_queue`。
>
> **Step 4（Controller → RabbitMQ → Car）**：Controller 向 `car.broadcast` Fanout 交换机广播 `"1"`（移动命令），所有小车队列收到消息。
>
> **Step 5（Car → Redis）**：每个 CarAgent 从自己的任务队列 `LPOP` 取一步，`SET ... NX EX 5` 原子抢占目标格子，`HSET` 更新位置，`SETBIT` 点亮 3×3 视野。
>
> **总结**：小车位置、地图数据走 Redis（黑板）；启停命令、导航任务、节拍广播走 RabbitMQ（消息总线）。

**由**：A（组长），B、C 可补充自己模块的细节

#### QA-04 为什么 Controller 必须是唯一的？

**Q**（王老师）：你们文档里说 Controller 只能运行一个实例，为什么这么设计？

**A**：
> Controller 必须是唯一的原因是它会广播移动命令。如果有两个 Controller 同时广播节拍，小车会收到双倍的命令，同一个 tick 内移动两步，导致：① 小车位置状态不一致；② 碰撞检测失效（两个 Controller 各自独立判断探索完成，可能一个判定完成一个不判定）；③ 录制帧和 tick 号对应不上。
>
> 理论上可以支持多个 Controller 通过选举机制（如 Raft）实现主备切换——但那对于课设场景过度设计了。我们只需要保证"同一时间只有一个 Controller 在工作"，所以用了 Redis 的 SETNX 分布式锁来实现。获得锁的 Controller 正常运行，拿不到锁的自动退出。

**由**：A（组长）

#### QA-05 分布式锁的具体实现？锁过期了怎么办？

**Q**（王老师）：你们说用了 Redis 分布式锁保证 Controller 单实例，能说下具体怎么实现的吗？如果 Controller 在处理 tick 时锁过期了怎么办？

**A**：
> 实现细节如下：
>
> **获取锁**：启动时执行 `SET controller:lock <instanceId> NX EX 30`。`NX` 保证原子性——如果 key 已存在则失败。`EX 30` 设 30 秒 TTL。
>
> **续约**：每 10 秒执行 Lua 脚本 `if GET(lock) == instanceId then EXPIRE(lock, 30) end`——只有锁的当前持有者才能续约，防止误续约。
>
> **释放**：JVM 关闭时通过 Shutdown hook 执行 Lua 脚本 `if GET(lock) == instanceId then DEL(lock) end`——避免异常退出后锁残留。
>
> 关于"锁过期"的问题：我们设计的 TTL（30 秒）远大于单次 tick 耗时（50-300ms）。即使极端情况下 tick 卡住了，续约线程每 10 秒检查一次，只要进程还在就会续约。如果进程真的崩溃了，锁最多 30 秒后自动释放——这个时间窗口是安全的，因为新 Controller 启动时 Redis 中的地图状态是完整的，它可以从当前状态继续调度。

**由**：A（组长）

#### QA-06 各模块部署为独立 JVM 进程的好处？

**Q**（陈老师）：你们把系统分成 4 个独立 JVM 进程（Controller/Navigator/Car/Recorder），比单进程有什么好处？

**A**：
> 主要三点好处：
>
> **第一，故障隔离。** 如果 Navigator 模块因为某个 BUG 导致内存溢出，只会影响它自己的进程，正在移动的小车和正在录制的 Recorder 完全不受影响。单进程中一个模块崩溃会导致整个系统宕机。
>
> **第二，独立扩缩容。** 我们可以为不同模块分配不同的 JVM 堆内存——Navigator 需要较多内存做路径搜索（-Xmx256m），Car 模块只需要很小的堆（-Xmx64m）——每个模块各取所需，不互相争抢。
>
> **第三，热升级。** 要升级路径算法时，只需要替换 navigator.jar 并重启 Navigator 进程，Controller 和 Car 继续运行不受影响——这在单进程架构中是做不到的。
>
> 代价是进程间通信有额外的序列化开销，但 Redis 本地的往返延迟 <1ms，完全可以接受。

**由**：A（组长）

#### QA-07 小车和导航器为什么要分两个模块？

**Q**（王老师）：导航器和小车为什么拆成两个模块？合在一起不也能工作吗？

**A**：
> 分开是因为它们的计算性质完全不同：
>
> **Navigator（路径规划）是 CPU 密集型**——A* 搜索需要大量计算。当小车数量增多时，路径规划请求也增多。所以我们给 Navigator 配置了 8 个 worker 线程池来并发处理。
>
> **Car（小车移动）是 I/O 密集型**——每次 tick 只做 Redis 读写操作（LPOP、SET NX EX、HSET、SETBIT），CPU 占用很低。瓶颈在 Redis 网络 I/O。
>
> 如果合在一起，当 Navigator 在做密集路径搜索时，会抢走 Car 处理 tick 的 CPU 时间，导致小车移动变慢。分离后，我们可以把 Navigator 和 Car 放在不同的 CPU 核心上运行，互不干扰。这也符合软件体系结构的"关注点分离"原则。

**由**：B（导航）+ C（小车）可分别补充

#### QA-08 为什么用 Fat JAR 不用 Spring Boot？

**Q**（陈老师）：你们用 Fat JAR 独立部署，为什么不用 Spring Boot？是不方便吗？

**A**：
> 用 Fat JAR + 纯 Java main 方法启动，而不是 Spring Boot，主要基于两个考虑：
>
> **第一，避免过度框架化。** 我们的系统只有 4 个后端模块，每个模块的功能很清晰——监听消息、读写 Redis、执行逻辑。Spring Boot 的自动配置、IoC 容器、内嵌 Tomcat 等功能我们都不需要，引入反而增加了启动时间和包体积。
>
> **第二，课设要求的"独立计算组件"。** 课设要求每个构件编译为独立可执行组件。使用 Spring Boot 打包也是 fat JAR，但会带来额外的框架依赖。我们通过 `maven-shade-plugin` 打包的 fat JAR 约 4.2MB，而 Spring Boot 打出的包至少 15MB+。
>
> 当前方案每个模块就是一个 `main()` 方法加几个核心类，逻辑清晰，老师评审代码时也容易阅读。

**由**：A（组长）

#### QA-09 多个 View 同时运行会不会互抢？

**Q**（陈老师）：你们说显示界面可以独立部署多个，那一个配置员在操作地图，另一个也在操作，会不会冲突？

**A**：
> 多个 View 实例会各自独立连接 Redis 和 RabbitMQ。如果两个配置员同时修改同一个数据（比如在同一位置放置不同的小车），后写入的会覆盖先写入的——Redis 本身没有事务保护跨键操作。
>
> 但在实际使用场景中，这个问题的风险不大：系统通常只有一个配置员在操作地图、放置小车和障碍物；分析员和管理员只是查看状态和录制回放，不会写地图数据。如果确实需要防止冲突，可以在前端加一个"编辑锁"——用 Redis 的 SETNX 实现，类似 Controller 的分布式锁——有人编辑时锁住，其他人只能查看。
>
> 不过目前的实现中，我们优先保证了"可独立部署"这个课设要求，多人编辑冲突问题通过使用场景约定来规避。

**由**：D（前端）

#### QA-10 如果 Car 进程崩溃了会怎样？

**Q**（王老师）：假设系统正在巡检，Car 进程突然崩溃了，已经移动的小车会怎样？

**A**：
> Car 进程崩溃意味着所有小车线程都停了。已经分配到任务队列中的路径数据仍然在 Redis 中（`{id}_task_queue`），小车的位置和状态也在 Redis 中（`Cars:{id}` hash）。
>
> 重启 Car 进程后，CarManager 会重新扫描 `Cars:*` 键，自动为每辆小车创建 CarAgent 线程并监听对应的 `car.no{id}` 队列。如果小车之前有剩余路径在队列中，它会在下一个 tick 到来时继续移动。
>
> 但 Controller 在 Car 崩溃期间会继续发送 tick 广播——这些消息会在 RabbitMQ 的 `car.no{id}` 队列中堆积。Car 重启后，它会消费队列中堆积的消息，但因为小车状态机判断（IDLE 或路径为空时跳过），多余的 tick 消息会被安全地忽略。所以整体上，Car 崩溃—重启是小车无损的。

**由**：C（小车）+ A（组长）

#### QA-11 你们怎么保证各个模块的启动顺序？

**Q**（陈老师）：各模块有依赖关系吗？启动顺序有要求吗？

**A**：
> 模块间存在弱启动依赖：
>
> - **必须最先启动**：RabbitMQ 和 Redis（基础设施）
> - **没有严格顺序**：Navigator、Car、Recorder 三个知识源可以任意顺序启动，它们各自声明自己的队列和交换器（幂等操作，多次声明不影响）

>
> 所以我们的一键启动脚本（Launcher）先启动 Navigator、Car、Recorder，间隔 500ms，最后启动 Controller。这样能保证在第一个 tick 到来时所有知识源都已就绪。

#### QA-12 你们的设计中有没有用设计模式？

**Q**（王老师）：能说说你们在代码中用了哪些设计模式？各自用在什么地方？

**A**：
> 主要用了以下几个设计模式：
>
> **策略模式（Strategy）**——三种路径算法（A*/双向A*/Dijkstra）都实现了 `PathFinder` 接口，Navigator 通过 Redis 中的 `Algorithm` 键运行时动态选择算法实现。这是最典型也最完整的策略模式应用。
>
> **单例模式（Singleton）**——每个模块的 `main()` 启动后，核心管理器（ControllerAgent、CarManager、RecorderService）都是单例运行。
>
> **模板方法模式（Template Method）**——`CarAgent.handleTick()` 定义了小车一步移动的算法骨架（检查→取路径→碰撞检测→移动→点亮视野），子步骤中的具体实现（如碰撞检测策略）可以独立变化。
>
> **工厂模式（Factory）**——`NavigationService.createFinder()` 根据算法编号创建对应的 PathFinder 实例。
>
> **黑板模式（Blackboard）**——这是架构级别的模式，Redis 是黑板，各知识源通过它共享数据。

**由**：A（组长）主答，B 可补充策略模式细节

**QA-12a: Launcher 一键启动是怎么实现的？**

**Q**（王老师）：你们的 Launcher 一键启动怎么做到启动四个独立的程序？

**A**：
> 使用 Java 的 `ProcessBuilder`——每个模块启动为独立的 JVM 进程。先声明 RabbitMQ 拓扑（交换机、队列），然后用 `ProcessBuilder` 分别启动 controller.jar、navigator.jar、car.jar、recorder.jar，间隔 500ms 防止瞬时资源争抢。

> JAR 路径通过四重回退机制定位：命令行参数 → `APP_HOME` 环境变量 → 自身 JAR 位置 → `user.dir`。JVM 关闭时 shutdown hook 会销毁所有子进程，确保优雅终止。

#### QA-13 小车寻路的时候是随机移动吗？

**Q**（陈老师，看着屏幕上的小车移动）：小车寻路的时候是随机移动还是有什么策略？（常见问题）

**A**：
> 小车不是随机移动的。路径规划由 Navigator 模块负责，使用**贪心最近目标策略**：
>
> 1. 每次导航时，从小车当前位置进行 BFS（广度优先搜索），找出曼哈顿距离最近的未探索且可达的格子作为目标
> 2. 然后用配置的路径算法（A*/双向A*/Dijkstra）搜索从当前位置到目标的最短路径
> 3. 如果所有可达格子都已探索，小车自动停止
>
> 我们之前确实尝试过随机选择目标，结果发现小车在地图上乱走，探索效率很低。改成贪心最近策略后，小车会优先探索附近的格子，整体探索速度提升了很多。

**由**：B（导航）

#### QA-14 像屏幕上这种小车到不了的地方怎么处理的？

**Q**（陈老师，指着被障碍物包围的区域）：你看这里，障碍物中间有一块区域，小车能进去吗？如果进不去怎么办？

**A**：
> 这是我们在开发中遇到的一个实际问题。对于被障碍物完全包围的区域，小车确实进不去，但我们有两个机制处理：
>
> **第一，BFS 可达性检测**。每次导航时，`TargetSelector` 会用 BFS 从小车当前位置出发，只选择"可达"的未探索格子作为目标。如果一块区域被障碍物包围、没有入口路径，BFS 就搜不到它，小车不会白费力气去尝试。
>
> **第二，探索完成时的自动判断**。Controller 使用 `Blackboard.isFullyExplored()` 检查——它会做一次 BFS，从所有小车当前位置出发，如果可达范围内所有格子都已探索，就判定探索结束。但如果存在被障碍物包围的不可达格子，BFS 搜索范围覆盖不到它们，系统会正常结束，通过 `exploration_result` 键标记为 `"partial"`（部分不可达）。前端会弹窗提示"可探索区域已全部巡检完毕，部分地块因被障碍物包围而无法进入"。

**由**：B（导航）

#### QA-15 三种路径算法有什么区别？哪个最好？

**Q**（王老师）：你们实现了 A*、双向 A*、Dijkstra 三种算法，它们有什么区别？实际用下来哪个效果最好？

**A**：
> 三者的核心区别在于是否使用启发式信息：
>
> **Dijkstra** 是最基础的算法，没有启发式，等权向外扩展，保证找到最短路径但扩展节点数最多。在 20×20 地图上差别不明显，但在更大地图上会显著变慢。
>
> **A*** 使用曼哈顿距离作为启发函数（h(n)），引导搜索方向偏向目标，扩展节点数远少于 Dijkstra。在我们的测试中，20×20 地图上 A* 平均搜索节点数约为 Dijkstra 的 1/5。
>
> **双向 A*** 从起点和终点同时做 A* 搜索，当两个方向的搜索相遇时结束。在大地图（50×50+）上效果最好，因为搜索空间从指数级降到两个半指数级。但在小地图上优势不明显，且代码复杂度最高。
>
> 综合来说，**A*** 最适合我们的场景（20×20 默认地图），效率和实现复杂度最均衡，所以作为默认算法。用户也可以在运行时通过前端下拉菜单切换到其他算法，不需要重启任何模块。

**由**：B（导航）

#### QA-16 导航器启动多个 worker 怎么分配任务？

**Q**（陈老师）：导航器默认 8 个 worker，Controller 怎么知道哪个 worker 是空闲的？

**A**：
> 每个 NavigationWorker 在 Redis 的 `navigator_status` hash 中维护自己的状态：`nav_1:working` 字段值为 `"true"` 或 `"false"`。Worker 的忙闲转换如下：
>
> - 空闲时：字段值 `"false"`
> - 收到导航任务后，立即设置为 `"true"`
> - 路径规划完成，写入路径队列后，恢复为 `"false"`
>
> Controller 的 `firstAvailableNavigator()` 方法在每 tick 轮询这个 hash——遍历 `nav_1` 到 `nav_N`，返回第一个值为 `"false"` 的 worker ID，然后通过 Direct Exchange 向对应的 `navigator.no{X}` 队列发送导航任务 JSON。
>
> 这是一个简单的**轮询机制**——先到先得，天然负载均衡。Worker 数量可通过环境变量 `NAVIGATOR_WORKER_COUNT` 配置，默认 8 个。

**由**：B（导航）+ A（组长补充 Controller 侧）

#### QA-17 选目标时为什么用 BFS 不用 A*？

**Q**（王老师）：你们找最近未探索目标用的是 BFS，但寻路用的是 A*，为什么目标选择不用 A*？

**A**：
> 因为目标选择的场景和路径搜索的场景不一样：
>
> **目标选择**需要"找到最近的未探索格子"，但**不知道目标在哪儿**——要在整个地图上搜索所有未探索格子，从中挑最近的一个。这本质上是"搜索 + 最近邻"问题，BFS 一层一层向外扩展，最先遇到的未探索格子就是最近的，天然匹配。
>
> **路径搜索**则是"已知起点和终点，找最短路径"——这是 A* 的典型场景。A* 利用启发函数引导搜索方向，效率高于 BFS。
>
> 如果用 A* 做目标选择，需要为每一个未探索格子跑一遍 A* 计算距离，复杂度是 O(未探索格数 × 路径长度)，比一次 BFS 的 O(地图格数) 高得多。

**由**：B（导航）

#### QA-18 GridMap 的代价为什么设已探索格子为 2、未探索格子为 1？

**Q**（王老师）：我看代码里 GridMap 的 moveCost，未探索格子代价 1，已探索格子代价 2——这跟直觉相反，为什么走过的地方代价更大？

**A**：
> 这是一个**引导性设计**，目的是让路径算法优先走未探索的格子。
>
> 如果所有格子代价相同，A* 默认走最短路径——最短路径往往会穿过已经走过的地方（因为已知的路径更直接），导致小车反复走老路，不去探索新区域。
>
> 我们把已探索格子的代价设为 2（未探索为 1），A* 在计算代价时就会发现"走未探索格子更便宜"，从而主动进入未探索区域探索。当小车要穿过大片已探索区去一个未探索目标时，路径仍然会走最短的已探索路——只是 A* 会"更愿意"拐进未探索格子里。
>
> 这个设计和"贪心最近目标"策略是互补的——一个从目标选择层面、一个从路径代价层面，共同引导小车高效探索。

#### QA-19 路径规划一次需要多长时间？

**Q**（陈老师）：Navigator 规划一条路径需要多久？会不会成为瓶颈？

#### QA-20 算法运行时切换是怎么实现的？

**Q**（陈老师）：你们说算法可以运行时热切换，具体怎么做的？

**A**：
> 实现原理很简单：算法选择不硬编码在代码里，而是通过 Redis 中的 `Algorithm` 键控制。

> 用户在配置界面下拉菜单选择算法时，前端会 `SET Algorithm "0"`（或 "1"/"2"）。Navigator 在 `NavigationService.createFinder()` 中每次都重新读取这个键：

>

>
> 切换是**逐任务生效**的——正在规划中的任务继续使用之前的算法，下一个任务自动使用新选择的算法。用户不需要重启任何进程，切换到新算法后下一次导航就会生效。

#### QA-21 小车只走上下左右四个方向，为什么不是八方向？

**Q**（陈老师）：我看小车移动是上下左右四方向的，为什么不是斜着走的八方向？

**A**：
> 因为我们选用的是网格地图的四邻域模型，小车每一步只能移动到上下左右相邻的格子。八方向（含对角线）虽然路径看起来更平滑，但会引入两个问题：
>
> **第一，碰撞检测更复杂。** 斜向移动时，两车"擦肩而过"的场景更难判断——比如 A 从 (0,0) 移动到 (1,1)，B 从 (1,0) 移动到 (0,1)，如果都走对角线就会在空中交换位置，这在实际物理场景中是不可能的。
>
> **第二，视野规则更复杂。** 我们现在的 3×3 视线阻挡检测已经针对四方向设计（检查对角时需要看两个正交邻格）。八方向移动会引入"斜向视野"的判断，规则复杂度翻倍。
>
> 四方向移动虽然看起来没八方向平滑，但对于课设的"探索覆盖"目标来说已经足够了——核心是格子被点亮，而不是路径的美观性。

#### QA-22 为什么 100 辆小车不能同时移动？

**Q**（王老师）：屏幕上几十辆车，它们是同时移动还是逐个移动？为什么不能完全同时？

**A**：
> 在同一个 tick 内，所有小车是**并发**但**不完全同时**移动的：
>
> 它们都收到同一个广播消息（`"1"`），各自的线程几乎同时开始执行 `handleTick()`。但"同时"受限于两个因素：
>
> **第一，Java 线程调度。** 每个 `CarAgent` 是一个独立的 Java 线程，由 JVM 线程调度器分配 CPU 时间片。几十个线程不可能在毫秒级完全同步运行。
>
> **第二，Redis 串行化。** 我们的碰撞避免机制使用了 `SET car_reserve:{x}:{y} NX EX 5` 原子操作。Redis 是单线程处理命令的，当多辆车同时 SET 抢占同一个格子时，Redis 会逐个处理——确保只有一辆车成功。这意味着实际上是小车按 Redis 的处理顺序依次"抢"目标格。
>
> **结论**：逻辑上是并发调度（同一 tick 内都收到了命令），物理上受线程调度和 Redis 单线程模型限制，做不到纳秒级的同时移动。但对于 500ms 的 tick 间隔来说，所有小车在 tick 结束时都能完成一步移动，用户看到的效果就是"一起动"。

**由**：C（小车）+ A（组长补充）

#### QA-23 小车为什么动的这么慢？

**Q**（陈老师，看着屏幕）：这个小车移动怎么感觉间隔好久才动一步？能调快吗？

**A**：
> 小车移动速度由 Controller 的 tick 间隔决定，默认是 **500ms**（即每秒 2 步）。这个速度看起来"慢"是故意的：
>
> **第一，可视化需要。** 如果 tick 间隔太快（比如 50ms），小车在屏幕上会瞬移，人眼根本跟不上，老师和评审都看不清小车是怎么移动的。500ms 一步适合观察。
>
> **第二，录制帧率对应。** 录制系统每 tick 保存一帧，太快的 tick 会导致录制的帧数据量过大，回放时也看不清楚。
>
> **但速度是可调的**——通过 `application.properties` 中的 `controller.tickMillis` 参数或 `CONTROLLER_TICK_MILLIS` 环境变量可以调整。比如设为 200ms 就变成每秒 5 步。实际上我们自己测试时经常设到 200ms 来加速验证。

#### QA-24 两车相撞怎么避免的？

**Q**（王老师）：你们怎么防止两辆小车撞到同一个格子里？

**A**：
> 我们使用**两阶段防护**：
>
> **阶段一：原子 SET NX EX 抢占（运行时防护）**
> 小车移动前，先执行 `SET car_reserve:{targetX}:{targetY} <carId> NX EX 5`。NX 表示仅当 key 不存在时才设置，EX 5 表示 5 秒后自动过期。整个过程是**单条 Redis 命令原子完成**——SET 和 EXPIRE 合二为一，不会出现"SET 成功但 EXPIRE 失败导致 key 永不过期"的竞态。Redis 是单线程处理命令的，即使两辆车同时尝试抢占同一格子，也只有一个能成功。
>
> **阶段二：哈希占用检查（双重校验）**
> 如果 SET NX EX 通过了，再检查目标格子是否已被其他小车的 `Cars:{id}` hash 占据——防止在 SET 成功到实际移动之间的微小时间窗口内，另一辆车也移动到了这个格子（虽然概率极低，但做双重防护更安全）。
>
> 此外，在导航阶段，Navigator 会**将其他小车的当前位置标记为临时障碍物**，让 A* 在规划路径时就避开。这样两个碰撞防护不冲突——导航层的静态避让是"路线规划"，SET NX EX 层的动态抢占是"最终执行"。

**由**：C（小车）

#### QA-24a 为什么用 SET NX EX 替代 SETNX+EXPIRE？

**Q**（王老师）：你们小车抢占目标格用的是 SET NX EX，为什么不直接用 SETNX 再 EXPIRE？

**A**：
> 这是一个关于**原子性**的改进。原来的实现用两步操作：先 `jedis.setnx(reserveKey, value)`，如果成功再 `jedis.expire(reserveKey, 5)`。这两步不是原子的——如果进程在 SETNX 成功后、EXPIRE 执行前崩溃（比如 JVM 被 kill -9 或机器断电），`car_reserve:{x}:{y}` 这个 key 就会**永久存在于 Redis 中**，导致该格子再也不能被任何小车抢占，地图上出现一个"死格子"。
>
> 改成 `jedis.set(reserveKey, value, SetParams.setParams().nx().ex(5))` 后，SETNX 和 EXPIRE 合并为**单条 Redis 命令**，原子执行。Redis 保证这条命令要么完整执行（设置值 + 设置 5 秒 TTL），要么完全不执行——不存在中间状态。这是 Redis 2.6.12+ 引入的 `SET key value NX EX seconds` 语法的 Jedis 封装。
>
> 即使没有进程崩溃的极端场景，两步非原子操作也存在微小的时间窗口——SETNX 成功后、EXPIRE 前的瞬间，如果该小车也正好因为其他原因退出了，key 同样会残留。原子 SET NX EX 从根本上消除了这个隐患。

**由**：C（小车）

#### QA-25 小车隔墙能看到东西吗？

**Q**（陈老师）：小车点亮周围格子的时候，如果有墙隔开了，还能看到墙后面的格子吗？

**A**：
> 不能。我们实现了**视线阻挡检测**来防止"隔墙点亮"。
>
> 3×3 视野中，上下左右四个正交方向无条件点亮——只要在界内且不是障碍物。但对角线方向的格子（左上、右上、左下、右下）需要额外检查：**只有当两侧的正交邻格中至少有一个不是障碍物时，才能点亮**。
>
> 举个例子：小车在 (x, y)，要判断能否点亮右上角 (x+1, y+1)。如果 (x+1, y) 和 (x, y+1) 这两个正交邻格**都是障碍物**，说明小车被墙角挡住了视线，(x+1, y+1) 就不会被点亮。
>
> 这个设计模拟了真实巡检机器人的传感器原理——只有"看得见"的格子才被计入已探索区域。

**由**：C（小车）

#### QA-26 巡检过程中还能不能放置小车？

**Q**（陈老师，常见问题）：系统正在巡检的时候，我能不能再放新的小车进去？

**A**：
> **可以。** 这是我们的一个特色功能——运行时动态添加小车，不需要停止巡检，也不需要重启任何进程。
>
> 实现原理：`CarManager` 每隔 1 秒执行 `SCAN Cars:*` 扫描 Redis 中的所有小车键。如果发现新的小车（`Cars:N` 键存在但还没有对应的 `CarAgent` 线程），它会自动创建一个新线程并启动监听 `car.no{N}` 队列。在下一个 tick 到来时，Controller 会扫描到新车是空闲状态，给它分配导航任务。
>
> 用户可以通过三种方式添加新车：
> 1. 前端配置界面点击"随机生成"输入数量
> 2. 点击地图格子手动放置
> 3. 直接在 Redis 中 `HSET Cars:6 x 10 y 10 state IDLE`
>
> 但注意：**不能删除正在巡检的小车**，因为它的任务队列中可能有未走完的路径——删除会导致路径丢失。

#### QA-27 小车状态机是怎样的？

**Q**（王老师）：能说说小车有哪些状态？各个状态之间怎么切换？

**A**：
> 小车有 4 种状态：
>
> - **IDLE**（空闲）：小车没有任务，等待 Controller 分配。Controller 扫描到 IDLE 且任务队列为空的小车时，会派发导航任务，将状态设为 ASSIGNING。
> - **ASSIGNING**（分配中）：NavigationWorker 正在为小车规划路径。路径写入任务队列后，小车状态保持 ASSIGNING。
> - **RUNNING**（运行中）：小车收到 tick 广播，执行一步移动。如果路径队列中还有剩余步数，继续保持 RUNNING；如果已走完最后一步，回到 IDLE。
> - **BLOCKED**（阻塞）：小车在 tick 中遇到障碍物（碰撞到其他车或障碍物），清空路径队列，回到 IDLE。
>
> 状态转换路径：`IDLE → ASSIGNING → (路径就绪) → RUNNING → IDLE / BLOCKED → IDLE`

**由**：C（小车）

#### QA-28 小车线程会不会占用太多资源？

**Q**（王老师）：每辆车一个 Java 线程，100 辆车就 100 个线程，会不会太多？

**A**：
> 这是个好问题。确实，100 个线程在资源上不算少，但有几方面理由说明这个设计在课设场景下是合理的：
>
> **第一，每个线程大部分时间在等待。** CarAgent 的核心循环是：阻塞等待 RabbitMQ 消息 → 收到后 5-15ms 处理 → 继续阻塞等待。500ms 的 tick 间隔中，线程真正工作的时间只有约 10ms，剩余 490ms 都在等待。所以线程实际占用的 CPU 非常少。
>
> **第二，我们设置了小车数量上限。** 默认上限是地图格数（20×20=400 辆），但实际使用的典型数量是 5-30 辆，线程压力完全可控。
>
> **第三，如果要优化**，可以用共享线程池 + NIO 模型（如 Reactor 模式）——用少量线程处理所有小车的 tick，但那样代码复杂度会急剧上升。对于课设项目，简单直接的每车一线程模型更易于理解和维护。我们也在模块说明中标注了这个约束条件。

**由**：C（小车）

#### QA-29 小车移动一步具体做了哪些事情？

**Q**（陈老师）：小车收到移动命令后到实际移动完成，中间做了哪些操作？

**A**：
> `handleTick()` 的执行顺序分解如下：
>
> 1. **状态过滤**：检查小车状态，如果是 IDLE 或 BLOCKED 直接跳过
> 2. **取路径**：`LPOP` 从任务队列取下一步坐标 `"x,y"`
> 3. **合法性验证**：检查目标格是否相邻（曼哈顿距离=1）且不是障碍物
> 4. **碰撞避免**：`SET car_reserve:{x}:{y} NX EX 5` 原子抢占目标格
> 5. **执行移动**：记录方向、`HSET` 更新坐标、删除 reserve key
> 6. **点亮视野**：`SETBIT` 点亮小车周围 3×3 区域（含视线检测）
> 7. **状态更新**：若路径已走完则设为 IDLE，否则保持 RUNNING
>
> 每一步都有微秒级耗时日志记录，方便性能分析。一次完整的 handleTick 约 5-15ms。

**由**：C（小车）

**QA-29a: 小车路径走完了任务队列空了怎么办？**

**Q**（陈老师）：小车走到路径最后一步后怎么办？

**A**：
> `handleTick` 在移动完成后检查 `LLEN {id}_task_queue`——如果为 0 说明路径走完，将状态设为 IDLE。Controller 在下一个 tick 扫描到该车 IDLE + 任务队列为空，会重新给它分配导航任务——BFS 找到下一个最近的未探索目标，规划新路径。

#### QA-30 录制是怎么保存探索过程的？

**Q**（陈老师）：录制的数据是怎么存的？每帧保存了什么？

**A**：
> 录制由 Recorder 模块负责。Controller 开始巡检时发送 `"1"` 到 `save.start` 队列，Recorder 收到后启动录制循环。
>
> 录制循环每 **tick（默认 500ms）** 保存一帧。每帧是一个 Redis Hash `Record:{fileNo}:{frame}`，包含一个 `snapshot` JSON 字段。JSON 内容为：
> - `MapView` 位图字节数组（Base64 编码）
> - `blockview` 位图字节数组
> - 所有小车的位置、方向、状态
> - 地图宽度和高度
>
> **数据结构**：
> - 元数据存在 `Save` hash（`file_num` 当前文件号、`order_view` 当前帧号计数）
> - 每帧数据存在 `Record:{N}:{F}` hash（N=文件号, F=帧号）
>
> 20×20 地图每帧约 1KB，一次完整探索（假设 400 tick）约 400KB，Redis 内存完全可以承受。

**由**：D（前端/回放）

#### QA-31 回放速度怎么控制的？

**Q**（陈老师）：回放时 1×、2×、3× 速度是怎么实现的？

**A**：
> 速度控制的核心是调整帧之间的播放间隔。
>
> Recorder 的 `playbackLoop()` 在逐帧恢复数据后调用 `delayMillis()` 方法。`delayMillis()` 从 Redis 的 `Save` hash 中读取 `triple_speed` 字段，计算间隔 = `max(100, 500 / triple_speed)`：
> - 1× 速度：间隔 = max(100, 500/1) = **500ms**（和录制时一样）
> - 2× 速度：间隔 = max(100, 500/2) = **250ms**（快一倍）
> - 3× 速度：间隔 = max(100, 500/3) = **166ms**（快三倍）
>
> 前端选择速度时写入 Redis `Save triple_speed`，Recorder 读取后自动调整。同时前端的分析员界面也有一个定时器，同步调整自己的进度条更新节奏，保持前后端一致。

**由**：D（前端/回放）

#### QA-32 回放进度条拖拽是怎么实现的？

**Q**（陈老师）：用户拖动回放进度条的时候，系统做了什么？

**A**：
> 进度条拖拽由前端和后端协同完成：

> **前端侧**：分析员界面有一个 `JSlider` 进度条，范围是 0 到 `Save.order_view`（总帧数-1）。用户拖动滑块触发 `stateChanged` 事件，暂停定时器，将目标帧号通过 RabbitMQ 发送给 Recorder（特殊回放命令）。

> **后端侧**：Recorder 收到跳转命令后，停止当前回放循环，读取 `Record:{fileNo}:{targetFrame}` 的 `snapshot` JSON，执行 `restore()` 方法——将帧数据中的 MapView、blockview、Cars:* 全部写回 Redis。

> **前端侧**：地图渲染组件每 200ms 刷新一次，写回 Redis 的数据被自动读取并渲染到屏幕上，用户就看到画面跳到了目标帧。
>
> 整个过程在毫秒级完成，用户感觉是"拖到哪就看到哪"。

#### QA-33 录制时系统崩溃了，录的数据还在吗？

**Q**（陈老师）：假设巡检到一半 Recorder 进程崩溃了，之前录的数据还在吗？

**A**：
> 在的。录制帧数据是写入了 Redis，Redis 默认是**持久化到磁盘**的（RDB 快照 + AOF 日志）。所以即使 Recorder 进程崩溃，已录制的帧数据不会丢失。
>
> 如果整个机器断电了——这取决于 Redis 的持久化配置。默认情况下 Redis 可能在最近一次 RDB 快照之后的数据会丢失几秒。但在课设的演示环境下，Redis 和所有模块运行在同一台机器上，不涉及网络分区或极端硬件故障，实际测试中数据都完整保存。
>
> 但注意：崩溃后元数据（`Save hash` 中的 `order_view` 计数）可能不完整。我们通过前端分析员界面的列表来展示可用的录制记录——如果帧数据不完整，播放时可能到某帧就停止了。

#### QA-34 回放和录制时能不能操作其他功能？

**Q**（陈老师）：回放过程中还能做其他操作吗？比如同时开始一次新的巡检？

> **录制时不能**：录制过程中 Controller 正在执行 tick，如果这时候配置员修改地图（改变障碍物布局或小车位置），会导致录制帧中记录的数据不一致——前一帧和后一帧的地图布局不同，回放时看起来会有"突变"。
>
> 所以我们做了限制：录制过程中，配置界面的"地图配置"和"放置障碍物"按钮会被禁用，防止误操作。但"查看状态"和"停止巡检"不受影响。

#### QA-37 地图渲染怎么做到几百毫秒刷新？

**Q**（王老师）：我看地图每 200ms 刷新一次，每次都从 Redis 读所有数据？不会很慢吗？

**A**：
> 200ms 刷新一次需要高效的数据读取。我们做了关键优化——**位图批量读取**：
>
> 优化前：前端逐格调用 `GETBIT MapView offset`——对于 20×20 地图（400 个格子），每次刷新需要 **400 次 Redis 网络请求**，每次约 0.5-1ms，总耗时约 200-400ms，加上 Swing 渲染时间，刷新间隔超过 1 秒。
>
> 优化后：前端一次调用 `GET MapView` 获取整个位图的字节数组（20×20=400bit=50 bytes），然后在本地的 Java 代码中按位解码：逐 byte → 逐 bit → boolean[][]。**1 次 Redis 网络请求替代了 400 次**，网络耗时从 200-400ms 降到 <1ms。Swing 渲染本身约 2-5ms，所以 200ms 刷新一次绰绰有余。
>
> 此外，`updateCacheFromRedis()` 开头现在有 `ensureConnection()` 调用——每次读取数据前先 PING 检测 Redis 连接是否存活，断连时自动从连接池获取新连接，防止长时间空闲后 Redis 断开导致前端画面卡住。
>
> 这个优化在 README 中记录为"从 O(N²) 次 Redis 调用降为 O(1) 次"，性能提升约 1000 倍。

**由**：D（前端）

#### QA-37a 前端怎么保证 Redis 连接不会断？

**Q**（陈老师）：前端界面长时间开着不用，Redis 会不会断开？断开后画面会不会卡死？

**A**：
> 这是我们在增速版中专门解决的问题。Redis 默认会对空闲连接做超时断开（`timeout` 配置项），如果前端长时间没有操作，底层 TCP 连接可能被 Redis 服务端关闭。一旦连接断开，后续的 `GET`/`SETBIT` 等操作都会抛异常，导致地图画面停止刷新。
>
> 我们在 6 个前端数据类（`MapModel`、`MapView`、`PlaybackModel`、`PlaybackMapView`、`UsersModel`、`UserModel`）中新增了**连接自动恢复机制**：
>
> - **`ensureConnection()`**：每次 Redis 操作前调用，执行 `PING` 命令检测连接是否存活。如果 PING 失败（连接已断开），自动调用 `renewConnection()` 从 JedisPool 连接池获取新连接，替换掉失效的旧连接。
> - **`renewConnection()`**：关闭旧连接 → 从连接池 `getResource()` 获取新连接。
>
> 这样即使 Redis 服务端主动断开了空闲连接，前端下一次刷新时（最多 200ms 延迟）会自动检测并恢复，用户无感知。这也保证了"多个 View 界面长时间不用再回来后依然正常工作"。

**由**：D（前端）

#### QA-39 配置员点击"开始巡检"后发生了什么？

**Q**（陈老师）：你能从头到尾讲一下点击"开始巡检"按钮后的完整流程吗？

**A**：
> 好的，从用户点击按钮到小车开始移动的全流程：
>
> **1. 前端写入配置**：点击按钮时，`MainController` 调用 `MapModel` 将当前配置（地图尺寸、小车位置、障碍物位置、算法选择）写入 Redis。
>
> **2. 发送启动命令**：通过 RabbitMQ 向 `controller` Fanout 交换机发送 `"1"`（启动命令）。Controller、Navigator、Recorder 都能收到这个消息。
>
> **3. Controller 启动 tick**：Controller 收到 `"1"`，调用 `startWork()` → 检查上一次探索是否完成（如果是则调用 `resetExplorationRun()` 重置运行态）→ 启动 tick 循环线程。
>
> **4. 第一个 tick 开始**：
>   - Controller 扫描 Redis 获取所有小车 ID
>   - 对每辆 IDLE 小车，轮询空闲 Navigator worker，发送导航任务 JSON
>   - Navigator worker 收到任务 → BFS 选目标 → A* 搜路径 → RPUSH 写入任务队列 → 设置小车状态
>
> **5. Controller 广播移动命令**：发送 `"1"` 到 `car.broadcast` Fanout 交换机，所有小车队列收到消息。
>
> **6. 小车移动**：每个 CarAgent 从任务队列 LPOP 取一步 → SET NX EX 抢占 → 更新位置 → 点亮视野。
>
> **7. 递归**：Controller 睡眠 500ms → 开始下一个 tick。
>
> **8. 并行录制**：Recorder 收到启动消息后，每个 tick 结束时保存一帧。
>
> 从点击按钮到第一辆小车开始移动，整体延迟约 500ms-1s（取决于是不是刚好赶上 tick 时机）。

#### QA-41a 多个 View 长时间不用会断连吗？

**Q**（陈老师）：假如答辩前我提前打开三四个 View 界面摆着，过了十几分钟再回来看，它们还会正常工作吗？

**A**：
> 会的。这得益于增速版中加入的 **自动连接恢复机制**——6 个前端数据类（`MapModel`、`MapView`、`PlaybackModel`、`PlaybackMapView`、`UsersModel`、`UserModel`）都在每次 Redis 操作前调用 `ensureConnection()`。
>
> 具体流程：地图每 200ms 刷新一次 → `updateCacheFromRedis()` 开头执行 `ensureConnection()` → 内部 `PING` 检测连接。如果 Redis 服务端因超时断开了 TCP 连接（典型场景是 `timeout` 配置为 300 秒），PING 会失败 → 自动从 JedisPool 连接池借一个新连接替换旧连接 → 后续操作正常执行。
>
> 整个过程耗时 <5ms，对 200ms 的刷新周期来说完全透明。用户放多久回来，画面依然在刷新，不会卡住、不会白屏、不会弹连接异常。状态面板的 2 秒轮询也同样有 `ensureConnection()` 保护。

#### QA-42 性能问题你们怎么解决的？

**Q**（王老师）：你们提到做了性能优化，能说说是怎么发现并解决性能瓶颈的？

**A**：
> 性能优化主要经历了三个阶段：
>
> **第一阶段：发现问题**
> 在开发中期，我们发现地图刷新非常慢（点击后等 2-3 秒才变），小车动一下要等很久。通过查看日志中每步的耗时分解，我们发现了主要瓶颈——前端地图渲染每次刷新都逐格调用 `GETBIT` 读取位图，20×20 地图需要 400 次 Redis 网络请求，加上 Swing 绘制，总耗时超过 2 秒。
>
> **第二阶段：定位根因**
> 对 CarAgent 的 `handleTick()` 各阶段做微秒级计时分解，发现：
> - 视野点亮阶段耗时最大（对 9 个格子各调用一次 SETBIT）
> - 但更大的瓶颈是前端渲染（2000ms+）
>
> **第三阶段：针对性优化**
> 做了三项优化：
> 1. **位图批量读取**（效果最显著）：一次 GET 获取整个位图 byte[] 在本地解码。前端渲染从 2000ms → 2ms。
> 2. **路径批量写入**：Navigator 用 RPUSH 一次写入整条路径（数组参数），替代逐条 LPUSH。
> 3. **设置 tick 日志**：每个 tick 输出耗时日志，持续监控性能变化。
>
> 最终效果：20×20 地图 + 5 辆车，单 tick 耗时稳定在 50-150ms。

#### QA-43 为什么不用多线程并发读 Redis？

**Q**（王老师）：你们每个模块只有一个 Jedis 连接，为什么不用连接池并发读？

**A**：
> 部分模块（如 View）已经使用了连接池——`RedisConnect` 类配置了 `maxTotal=50` 的连接池。原因如下：
>
> - Controller 模块：单线程 tick 循环，一次只做一个操作，单连接就够了
> - Navigator：8 个 worker 各自有独立 Jedis 连接（每个 worker 有自己的连接）
> - Car：每辆车的 CarAgent 在 tick 中串行执行 Redis 操作（5-15ms），单连接足够
> - Recorder：录制循环也是串行操作
> - **View 前端**：地图渲染、状态面板、用户操作可能并发，所以用了连接池（maxTotal=50）
>
> 另外，Redis 本身是单线程处理命令的，多连接并发读不会提高处理速度——只是减少客户端排队等待的时间。对于每个模块几十毫秒的操作量级，单连接足够了。

**由**：C（数据层）

#### QA-44 500ms tick 间隔怎么来的？能调吗？

**Q**（陈老师）：tick 间隔 500ms 是拍脑袋定的还是有依据？为什么不是 1 秒或者 100ms？

**A**：
> 500ms 是一个**权衡值**，考虑了三个因素：
>
> **可视化需求**：100ms 的 tick 间隔下，小车在屏幕上几乎是瞬移的，人眼无法追踪。500ms 一步能看到小车的移动路径。
>
> **Navigator 处理时间**：如果 tick 太短（<200ms），Controller 发送导航任务后还没来得及得到 Navigator 的响应就要广播移动命令了——小车拿到路径但还没规划完。500ms 给 Navigator 留了充足的余量。
>
> **录制帧率**：500ms 一帧（2fps）回放时看起来流畅。如果降到 1 秒一帧，回放时就一跳一跳的。
>
> 但这个值是**可配置的**——通过 `CONTROLLER_TICK_MILLIS` 环境变量可以在不重启 Controller 的情况下调整。我们在测试时经常调到 200ms 来加快验证速度。

**由**：A（组长）

#### QA-45 小车数量增多时性能下降严重吗？

**Q**（王老师）：你把小车从 5 辆加到 50 辆，性能会下降多少？有没有测试数据？

#### QA-45a 增速版做了哪些后端性能优化？

**Q**（王老师）：增速版 1.0 具体做了哪些后端性能优化？有什么量化效果？

**A**：
> 增速版做了三项后端性能优化，全部集中在 `Blackboard.java` 的底层数据操作：
>
> **1. SCAN 批量扫描提速**
> - 优化前：`SCAN cursor MATCH Cars:* COUNT 20`——每次 SCAN 只返回约 20 个 key，小车多时需要多轮迭代才能拿到全部 ID
> - 优化后：`SCAN cursor MATCH Cars:* COUNT 500`——COUNT 从 20 提升到 500，绝大多数场景一次 SCAN 即可返回全部键，减少 SCAN 迭代次数
>
> **2. 小车坐标批量读取（hmget 替代两次 hget）**
> - 优化前：`getCarPoint()` 分别执行 `hget(key, "x")` 和 `hget(key, "y")`——两次独立的 Redis 往返
> - 优化后：改用 `hmget(key, "x", "y")`——一条命令同时获取 x 和 y，Redis 往返从 2 次降为 1 次
> - 效果：100 辆车每 tick 都查询位置时，节省约 100 次 Redis 往返
>
> **3. 移除多余读操作**
> - 优化前：`setCarState()` 更新小车状态后，又执行 `hget` 把刚写入的值读回来做日志打印——一次多余的 Redis 读操作
> - 优化后：移除调试日志中多余的 `hget` 回读，直接使用本地已知的值构造日志
>
> 三项优化累计减少了大量不必要的 Redis 往返——在高并发场景（50+ 辆车）下尤为明显，单 tick 后端耗时降低约 15-25%。

#### QA-47 为什么不用数据库存数据？

**Q**（陈老师）：录制帧数据为什么用 Redis 存而不是 MySQL 这类关系数据库？

**A**：
> 主要是**数据结构匹配**和**性能**两个原因：
>
> **第一，我们的数据是键值对和位图结构**——地图是位图（byte[]）、小车是 Hash、路径是 List，这些天然匹配 Redis 的数据类型。如果用 MySQL 存位图，需要转换成 BLOB 字段，读写都要走 SQL 解析，多了不必要的开销。
>
> **第二，性能要求不同。** 录制帧每 500ms 写入一次，回放时每 500ms 读取一次——MySQL 也能做到这个频率，但每帧都要建立数据库连接、执行 SQL、关闭连接，代码复杂度大增。
>
> **第三，开发期便利性。** 用 Redis 可以直接通过 `redis-cli` 命令行检查数据状态——`GET MapView`、`HGETALL Cars:1`——调试效率极高。MySQL 需要额外的数据库管理工具。
>
> 所以我们的设计是：**运行时数据全走 Redis**，不做持久化到 MySQL。Redis 本身也支持 RDB/AOF 持久化，重启后数据不丢失。

**由**：C（数据层）+ A（架构）

**QA-47a: 为什么选择用 Redis Bitmap 存地图而不是 Hash 存每个格子？**

**Q**（王老师）：为什么用 Bitmap 存地图？用 Hash 每个格子一个字段不行吗？

**A**：
> Bitmap 是最紧凑的存储方式。20×20=400bit=50 bytes。如果用 Hash 存，每个格子一个字段需要 `field=offset, value=0/1`，400 个字段的 Hash 约 10KB+，是 Bitmap 的 200 倍。
>
> 更重要的是读取效率——Bitmap 一次 `GET key` 50 字节全部读回来，Hash 需要 `HGETALL` 返回 400 个字段，解码开销大得多。同时 Bitmap 的 `SETBIT` 操作是 O(1) 原子操作，比 Hash 的 `HSET field value` 语义更清晰。

**由**：C（数据层）

#### QA-51 为什么导航器需要多个 worker？

**Q**（王老师，对应要求 #4）：你们说导航器可以启动多个 worker，一个 worker 不行吗？

**A**：
> 一个 worker 在技术上可以工作，但会有性能瓶颈：
>
> 当有多辆空闲小车需要导航时，Controller 会逐个发送导航任务到唯一的 `navigator.no1` 队列。Worker 处理每个任务时需要：读取 Redis 位图 → BFS 选目标 → A* 搜路径 → RPUSH 写队列。如果单次规划耗时 5ms，5 辆小车就是 25ms，50 辆就是 250ms——接近 tick 间隔的一半，会导致小车等待导航的时间变长。
>
> 多 worker 模式下，每个 worker 监听独立队列（`navigator.no1~noN`），Controller 轮询空闲 worker 并行分配任务。8 个 worker 可以同时处理 8 辆车的路径规划，总吞吐量是单 worker 的 8 倍。
>
> Worker 数量通过 `NAVIGATOR_WORKER_COUNT` 配置，默认 8 个，可以根据地图大小和小车数量调整。

#### QA-53 控制器单实例怎么保证的？

**Q**（王老师，对应要求 #6）：我现在同时启动两个 Controller 会怎样？

**A**（模拟）：
> 我们需要启动第二个 Controller 来演示。（面向屏幕操作）
>
> 第一个 Controller 已经启动成功了。现在我在另一个终端再启动一个 Controller：
>
> ```
> java -jar controller.jar
> ```
>
> 看日志输出了——"Controller 实例已在运行，退出。"然后进程自动退出了。
>
> 原因是启动时执行了 `SETNX controller:lock <instanceId> EX 30`——Redis 中已经有第一个 Controller 创建的锁了，SETNX 返回失败，第二个 Controller 就认为锁已被占用，立即 exit(1)。

> 如果第一个 Controller 崩溃了——锁最长 30 秒后自动释放（TTL），新 Controller 可以获取到锁并正常工作。如果第一个进程正常退出——Shutdown hook 会执行 Lua 脚本释放锁，新 Controller 可以立即启动。

#### QA-54 远程部署时组件怎么通信？

**Q**（王老师，对应要求 #2、#5）：你说组件可以独立部署，那如果把 Redis 和 RabbitMQ 放在一台机器上，View 部署在另一台机器上，怎么让它们连上？

**A**：
> 前端 View 的两个连接工具类——`RedisConnect.java` 和 `StartProducer.java`——支持通过环境变量指定远程地址：

#### QA-56 多主机部署时数据怎么同步？

**Q**（王老师）：四台主机看到的数据怎么保持一致的？

**A**：
> 数据一致性的关键是我们用了**集中式共享黑板**——Redis。所有 View 实例都连接到主机 A 的同一个 Redis 实例（同一 DB 9），读取的是同一份数据：同一张 `MapView` 位图、同一个 `Cars:*` hash 集、同一套 `Save`/`Record:*:*` 回放数据。

>
> RabbitMQ 消息总线同理——所有 View 的 `StartProducer` 通过环境变量连接到主机 A 的 RabbitMQ，发送的启停命令统一被 Controller/Recorder 消费。

#### QA-58 组件之间怎么共享数据的？

**Q**（陈老师，对应要求 #7）：你解释下各组件之间数据是怎么共享的？能给个数据流的具体例子吗？

**A**：
> 以"小车更新位置并点亮视野"这个操作为例，完整的共享过程如下：
>
> **Car** 每步移动后，通过 `HSET Cars:1 x 5 y 8 direction 2` 和 `SETBIT MapView 5,8 1` 将位置和视野写入 Redis（黑板）。
>
> **View**（前端地图渲染）每 200ms 执行 `GET MapView` 获取位图、`HGETALL Cars:1` 获取小车位置。这些数据不会直接推送——而是 View 主动从 Redis 拉取。
>
> **Controller** 每 tick 执行 `SCAN Cars:*` 获取所有小车，检查 `*_task_queue` 长度判断是否需要分配导航任务。
>
> **Navigator** 在路径规划时读取 `GET MapView` 和 `GET blockview` 获取地图状态，然后通过 `RPUSH 1_task_queue` 写入路径。
>
> **Recorder** 在录制时读取所有数据保存为快照，回放时写回 Redis。
>
> 核心思想：Redis 是所有数据的"唯一真相来源"（Single Source of Truth），每个组件只通过 Redis 共享数据，不互相直接调用。

#### QA-59 组件独立性体现在哪？

**Q**（王老师，对应要求 #8）：你如何证明组件之间是高度独立的？

**A**：
> 从三个层面证明：
>
> **代码层面**：四个后端模块（Controller、Navigator、Car、Recorder）之间没有 `import` 依赖关系。它们唯一的公共依赖是 common 模块（Redis 键定义、消息模型）。但通过 fat JAR 打包，common 的代码被各自打包进各自的 JAR——运行时不存在代码级耦合。
>
> **进程层面**：每个模块是独立的 JVM 进程，拥有独立的堆内存、类加载器、线程池。A 进程的 Full GC 不会导致 B 进程暂停。A 进程的崩溃不会影响 B 进程的运行。
>
> **数据层面**：模块间只通过 Redis（数据）和 RabbitMQ（消息）通信。Controller 不持有 Car 的引用，Car 不调用 Navigator 的方法。它们甚至不知道彼此的存在——只知道自己"读什么键、写什么键、监听什么队列"。
>
> 我们可以现场演示：停掉 Navigator 进程，Controller 和 Car 的日志不会报任何错误——Controller 只是发现所有 Navigator worker 都忙，跳过导航任务派发。重启 Navigator 后，它自动恢复工作。

**由**：A（组长）

#### QA-60 组件易于升级怎么设计的？

**Q**（陈老师，对应要求 #9）：如果我要升级路径规划算法，怎么做？

**A**：
> 只需要替换 Navigator 模块的 JAR 文件并重启该进程。
>
> 具体步骤：
> 1. 在 navigator 模块中修改代码、添加新算法
> 2. 执行 `mvn -pl navigator clean package -DskipTests`（只编译 navigator 模块）
> 3. 将生成的 `navigator.jar` 复制到部署目录覆盖旧文件
> 4. 找到 Navigator 进程，Ctrl+C 停止
> 5. 重新执行 `java -jar navigator.jar` 启动
>
> 整个过程 **Controller、Car、Recorder 完全不需要停止或重启**。Controller 会继续发送导航任务到 MQ 队列，Navigator 重启后会消费队列中堆积的任务。升级期间任务的短暂延迟不影响系统整体运行。
>
> 同理，如果只是增加新的路径算法实现（新增一个继承 PathFinder 接口的类），连进程都不需要重启——通过 Redis 的 `Algorithm` 键运行时切换算法即可生效。

#### QA-61 你们做了哪些性能优化？

**Q**（王老师，对应要求 #10）：你们在"保持独立性前提下提高运行效率"方面做了哪些工作？有数据支撑吗？

**A**：
> 我们做了以下主要优化：
>
> **1. 位图批量读取（前端渲染）**
> - 优化前：400 次 GETBIT → 2000ms
> - 优化后：1 次 GET → 2ms
> - 方法：一次拉取整个位图 byte[]，本地逐 bit 解码
>
> **2. 路径批量写入（导航器输出）**
> - 优化前：逐条 LPUSH 路径点
> - 优化后：一条 RPUSH 写入整条路径（变长参数）
> - 效果：路径写入时间从 O(N) 次 Redis 调用降到 O(1)
>
> **3. 导航 Worker 池**
> - 8 个 worker 并发处理多车导航任务
> - 比单 worker 吞吐量提升约 8 倍
>
> **4. 本地缓存 BFS**
> - `Blackboard.isFullyExplored()` 使用本地 byte[] 执行 BFS
> - 零 Redis 调用完成可达性检测
>
> **5. Common 模块共享**
> 所有模块共用 common 中的 Redis/RabbitMQ 工具类和键定义——不重复造轮子，减少代码量和维护成本。
>
> 最终效果：单 tick 耗时从最初的 500ms+ 降至 50-150ms，前端渲染从 2000ms 降至 <5ms。

#### QA-65 系统怎么正常关闭？

**Q**（陈老师）：演示结束怎么关系统？直接关窗口就行吗？

**A**：

> 如果手动独立启动各模块，只需在各终端按 Ctrl+C——每个模块都注册了 JVM Shutdown hook，会优雅关闭 Redis 连接和 RabbitMQ 连接，释放分布式锁等资源。

#### QA-66 系统可以扩展什么新功能？

#### QA-67 和市面上同类系统比有什么优缺点？

**Q**（陈老师）：你们这个系统跟真正的变电站巡检系统比起来，有哪些优点和不足？

**A**：

> 1. **架构清晰**：黑板架构确保了组件间的高度独立性和可扩展性，这在工业级的单体应用中并不常见
> 2. **功能完整**：从地图配置、多车调度、碰撞避免到录制回放，形成了一个完整的功能闭环
> 3. **可观测性好**：日志系统和状态面板让系统状态完全透明，方便调试和监控

> 1. **没有真实硬件接口**：当前是小车在屏幕上移动的仿真，真实巡检系统需要对接机器人硬件（传感器数据、机械臂控制、充电管理）

### DEMO-05 算法切换对比

**老师要求**："三种算法有什么不同？能不能现场切换看看？"

   > 运行约 10 秒后停止。

   > B："总结一下：A* 综合最优作为默认，双向 A* 适合大地图，Dijkstra 适合不需要启发式的通用场景。切换不需要重启任何进程——Navigator 在每个任务开始前重新读取 Algorithm 键即可生效。"

   > B："启动巡检后，小车会探索房间外的所有可达格子。房间内的那 1 个格子虽然在未探索状态，但小车无法绕过障碍物到达它。"

   > B："Redis 的 `exploration_result` 键会标记为 `'partial'`（部分不可达），区别于正常全部可探索的 `'complete'`。"

### DEMO-15 Launcher 一键启动 vs 独立启动对比

**老师要求**："你演示一下一键启动和分别独立启动两种方式。"

**操作人**：A  
**解说人**：A

**操作步骤**：

1. **演示一键启动**（A 操作 + A 解说）
   > A 先确保环境干净（运行 clean-runtime），然后在终端执行 `java -jar launcher.jar`。
   > A："看——一个命令启动了全部四个后端模块。Launcher 控制台会显示各模块的启动日志和 PID。它们各自是独立的 JVM 进程，但生命周期由 Launcher 统一管理——Launcher 进程退出时，会通过 shutdown hook 销毁所有子进程。"

2. **停止一键启动**（A 操作）
   > A Ctrl+C 停止 Launcher。
   > A："Launcher 停止后，所有四个后端进程也都停止了。"

3. **演示独立启动**（A 操作 + A 解说）
   > A 依次开 4 个终端，分别执行：
   > ```
   > 终端1: java -jar controller.jar
   > 终端2: java -jar navigator.jar
   > 终端3: java -jar car.jar
   > 终端4: java -jar recorder.jar
   > ```
   > A："独立启动方式适合演示各模块的独立性——Controller、Navigator、Car、Recorder 各跑各的，随时可以单独停止一个（Ctrl+C）展示故障隔离效果。"

4. **对比说明**（A 解说）
   > A："总结两种方式的适用场景："
   > > | 方式 | 命令 | 适用场景 |
   > > |------|------|---------|
   > > | Launcher 一键启动 | `java -jar launcher.jar` | 日常开发、快速演示、统一生命周期管理 |
   > > | 独立启动 | `java -jar xxx.jar` | 演示组件独立性、故障恢复、逐个升级 |
   > >
   > > 我们现在实际使用的启动脚本（`start.bat` → `menu.ps1` → `run-all.ps1`）是对 Launcher 方式的封装——自动完成构建、初始化、清理、启动全流程——但底层仍然是启动四个独立 JVM 进程，和上面的原理一致。

### 答辩流程建议

建议按以下流程组织整个答辩（约 15-20 分钟）：

| 时间段 | 内容 | 主讲人 | 说明 |
|-------|------|--------|------|
| 0:00-2:00 | 系统概述 + 架构讲解（放架构图 PPT 或纸质稿） | A（组长） | 说清楚黑板架构、组件划分、技术栈 |
| 2:00-5:00 | 标准巡检演示（DEMO-01） | D 操作，A 解说 | 展示最核心功能 |
| 5:00-7:00 | 路径规划 + 算法切换（DEMO-05） | D 操作，B 解说 | 展示算法能力 |
| 7:00-9:00 | 碰撞避免 + 车道动（DEMO-03/04） | D 操作，C 解说 | 展示小车细节 |
| 9:00-11:00 | 录制回放演示（DEMO-06） | D 操作，D 解说 | 展示回放能力 |
| 11:00-12:00 | 组件独立性演示（DEMO-08/09） | A 操作 | 快速展示 |
| 12:00-15:00 | 老师提问环节 | 全员 | 按分工回答 |
| 15:00+ | 自由交流 | 全员 | 展示其他功能 |

### 回答技巧

- **不要抢答**：老师提问时，先让对应负责的成员回答，其他人补充
- **承认不足**：如果问到确实没有的功能或实现不够好的地方，坦诚说明"我们在课设范围内做了 XXX，这个功能我们了解过但没时间实现了"
- **不说谎**：绝对不要编造不存在的数据或功能，被追问时圆不回来
- **对接 PPT**：如果答辩需要 PPT，本指南中的 QA 和 DEMO 可以直接作为 PPT 的"可能提问"参考
- **组长控场**：如果某个成员回答不上来，组长主动接过话头补充，不要冷场

### 老师常见追问方向

| 老师 | 偏好方向 | 典型追问 |
|------|---------|---------|
| 陈老师（软件工程） | 功能完整性、课设要求满足度、代码结构 | 课设要求逐项验证、功能是否齐全、代码是否规范 |
| 王老师（软件体系结构） | 架构设计选择、设计模式、组件通信 | 为什么选这个架构、组件独立性怎么保证、数据流画一下 |
